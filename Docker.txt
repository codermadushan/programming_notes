---------- Introduction --------------------------------------------------------

* Docker is a container technology: A tool for creating and managing containers.


* Container:

      * A standardized unit of software.

      * A package of code and dependencies to run that code.


* Why we need containers?

      1. Environment consistency:

          * Same OS, runtime, and dependencies everywhere.


      2. dependency isolation:

          * Each project has its own environment.
          * No conflicts between libraries, runtimes, or DB versions.


      3. Fast project setup:

          * Start full stack with one command (docker-compose up).
          * Easy onboarding for new developers.


      4. Lightweight execution:

          * Faster and smaller than virtual machines.
          * Uses host OS kernel (no full OS per app).


      5. Simplified deployment:

          * Build once -> run anywhere.
          * Same container runs in dev, staging, and production.


      6. Reproducibility & Versioning:

          * Container images are immutable.
          * Easy rollback to previous versions.


      7. Microservices support:

          * Run multiple services independently.
          * Scale specific services easily.


      8. Better resource utilization:

          * Efficient CPU and memory usage.
          * Run many containers on one machine.


      9. Improved isolation & security:

          * Applications run in isolated environments.
          * Reduces risk of system-wide issues.


      10. Works with orchestration tools:

          * Easily managed at scale using -> Kubernetes.


* To check disk usage:

    docker system df


* Full cleanup:

    docker system prune -a --volumes

---------- Images vs Containers ------------------------------------------------

* A Docker image is a read-only template that contains everything needed to run
  an application.

    Includes:

        * Application code
        * Runtime (e.g., Node.js, Python)
        * Libraries & dependencies
        * Configuration

    * Think of an image as a blueprint.




* A container is a running instance of an image.

    Key properties:

        * Has a read-write layer on top of the image
        * Is isolated (process, filesystem, network)
        * Can be started, stopped, and deleted

    * Think of a container as a live application created from the blueprint.




* One image     → can create many containers
* One container → uses exactly one image




* Analogy

    * Image = Class
    * Container = Object (instance of the class)




* We can consider a container as follows:

    +-------------------------------------------------+
    |                                                 |
    |  Container Layer (Read-Write)                   | <--- Container
    |                                                 |
    |  +---------------------------------+            |
    |  |                                 |            |
    |  |  Instruction #3: Image Layer 3  |            |
    |  |                                 |            |
    |  |  Instruction #2: Image Layer 2  | <--- Image |
    |  |                                 |            |
    |  |  Instruction #3: Image Layer 3  |            |
    |  |                                 |            |
    |  +---------------------------------+            |
    |                                                 |
    +-------------------------------------------------+

---------- Dockerfile ----------------------------------------------------------

* A Dockerfile is a text file that contains a set of instructions used to build
  a Docker image.


* Docker reads the Dockerfile and executes each instruction step-by-step to
  create an image


* Dockerfile → builds → Image → runs → Container


* Common instructions

    FROM       → base image

    WORKDIR    → set working directory (default: .)

    COPY / ADD → copy files into image

    RUN        → execute commands during build

    CMD        → default command when container starts

    EXPOSE     → declare ports


* Each instruction in a Dockerfile creates a seperate layer.


* Layers are cached by Docker to speed up builds.


* If a layer has not changed, Docker reuses it from cache.


* If a layer is rebuilt:

    * That layer is rebuilt.

    * All sebsequent layers are also rebuilt (cache invalidated).

    Ex:

        FROM node

        WORKDIR /app

        COPY . /app

        RUN npm install

        EXPOSE 80

        CMD [ "node", "server.js" ]


    * In this case, if we change our source code, the "COPY . /app" should be
      rebuilt.

    * Therfore all layers after that command is also going to be rebuilt.

    * But "RUN npm install" is an unnecessary rebuild.

    * Therefore we can optimize this as follow:

        FROM node

        WORKDIR /app

        COPY package.json /app

        RUN npm install

        COPY . /app

        EXPOSE 80

        CMD [ "node", "server.js" ]

---------- Basic Image Commands ------------------------------------------------

* Docker images are read-only. If you make any changes to the source code, you
  have to rebuild the image.




* Show all images:

    docker images -a




* Build an image:

    docker build -t <name:version> <Dockerfile_path>




* Rename an image:

    docker tag <old_name:old_version> <new_name:new_version>


    * A new image will be created with the new name




* Run an image (Create a container):

    docker run -p <host_port:container_port> <name:version>


        * Use -d to run the container in detach mode.

        * Use --rm command to remove the container after stopping it.

        * Use --name <container_name> to provide a name to the container.




* Remove an image:

    docker rmi <name:version>

--------------------------------------------------------------------------------




---------- Basic Containers Commands -------------------------------------------

* Show containers:

    docker ps    : show all running containers

    docker ps -a : show all containers




* Stop a container:

    docker stop <container_name>




* Start a stopped container:

    docker start <container_name>    : In detach mode

    docker start -a <container_name> : In attach mode




* Remove a container:

    docker rm <container_name>

--------------------------------------------------------------------------------




---------- Attached & Detached Containers --------------------------------------

* When we create and run a container using an image, we can use -d flag to
  detach from the container:

    docker run -d ...


* To attach again to see future log messages:

    docker attach <container_name>


* To see log messages without attaching:

    docker logs <container_name>


* To see previous log messages and attach again to see future logs:

    docker logs -f <container_name> (-f -> follow)

--------------------------------------------------------------------------------




---------- Interactive Mode ----------------------------------------------------

* If you want to input data (STDIN), we have to use Interactive mode.


* Create a container in interactive mode:

    docker run -it ...

        -i : Interactive mode
        -t : Create a terminal to enter inputs




* To start a stopped container:

    docker start -ia ...

        i : Interactive mode
        a : Attach mode

--------------------------------------------------------------------------------




---------- Remote Repository Of Docker Images ----------------------------------

* To upload an image to the DockerHub, it should contains the tag as follows:

    <username>/<repository_name>:<version>

    * To rename an existing image:

        docker tag <old_image_name>:<old_tag> <username>/<new_image_name>:<tag>




* To login use:

    docker login




* To logout use:

    docker logout




* To upload an image to DockerHub:

    docker push <image>




* To get an image from the DockerHub:

    docker pull <image>

---------- Managing Data & Working With Volumes --------------------------------

* Data:

    1. Application Data:

            * Ex: Code + Environment

            * Written & provided by you (=the developer).

            * Added to image in build phase.

            * Fixed: Can't be changed once image is built.

            * Read-only, hence stored in images.


    2. Temporary App Data:

            * Ex: Entered user inputs.

            * Fetched/Produced in running container.

            * Stored in memory or temporary files.

            * Dynamic and changing, but cleared regularly.

            * Read + Write, Temporary, hence stored in containers.


    3. Permanent App Data:

            * Ex: User accounts.

            * Fetched/Produced in running containers.

            * Stored in files or a database.

            * Must not be lost if container stops/restarts.

            * Read + Write, Permanent, stored with containers & volumes.




* Suppose we run a server inside a container. In this case we store user data of
  the server as txt files in the container file system.

* When you stop the container and start it again, these stored files are still
  exist because of the container's read-write layer.

* But if you remove the container, data will be removed.

* That is why we need volumes.




---------- (Volumes) ----------

* Volumes are folders in your host machine hard drive which are mounted into
  containers.


* Volumes persist if a container shuts down. If a container restart and mounts a
  volume, any data inside of that volume is available in the container.


* A container can write data into a volume and read data from it.


* When a container writes -> data goes directly into the volume (host storage).


* No "sync later" - It's immediate persistence.


* There are 2 types of docker volumes:

    1. Named Volumes
    2. Bind Mounts


* Anonymous volumes are a third form of docker volume - but conceptually they're
  a variant of named volumes without an explicit names.

        * To specify anonymous volumes in Dockerfile:

            VOLUME [ "/app/feedback" ]

        * Anonymous volumes are tied to the specific container they were created
          with.

        * When you remove a container with "docker rm" command, the link between
          the container and the volume is broken.

        * To remove a container with its volumes:

            docker rm -v <container_name>


* To see all volumes:

    docker volume ls


* To remove all unused volumes:

    docker volume prune    : Remove all unused anonymous volumes

    docker volume prune -a : Remove all unused anonymous + named volumes


* Remove a specific volume:

    docker volume rm <volume name>




---------- (Named Volumes) ----------

* In Docker, a named volume is a persistent storage mechanism managed by Docker
  itself, where you explicitly give the volume a name instead of letting Docker
  generate one automatically.


* You cannot defined named volumes inside the Dockerfile.


* Define a named volume:

    docker run -v <volume_name>:<container_file_path> ...


* To remove all unused named volumes:

    docker volume prune -a




---------- (Bind Mounts) ----------

* A way to map a specific directory or file from your host machine into a
  container. Instead of docker managing the storage (like volumes), you directly
  "bind" a host path to a container path.


* Changes on host -> Immediately visible in container.


* Mostly used in development to mount source code.

    docker run -v $(pwd):/app <container_name>
